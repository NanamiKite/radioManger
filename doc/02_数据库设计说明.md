# 数据库设计说明文档

## 概述

RadioManager采用MySQL 8.0+作为关系数据库，使用SQLAlchemy ORM进行数据映射。系统支持三种部署模式下的数据持久化方案。

## 数据库架构

### 数据库基本信息
- **DBMS**：MySQL 8.0+
- **字符集**：UTF8MB4（支持中文和emoji）
- **ORM框架**：SQLAlchemy
- **迁移工具**：Alembic

### 表设计原则
1. 所有表都包含创建时间和更新时间
2. 主键使用UUID或自增INT
3. 外键关系使用ON DELETE CASCADE
4. 关键字段创建索引以优化查询性能
5. 使用逻辑删除（soft delete）保留数据追踪

## 核心数据表

### 1. users（用户表）
用户账户信息和认证数据

```sql
CREATE TABLE users (
    id INT AUTO_INCREMENT PRIMARY KEY COMMENT '用户ID',
    username VARCHAR(50) UNIQUE NOT NULL COMMENT '用户名',
    email VARCHAR(100) UNIQUE NOT NULL COMMENT '邮箱',
    password_hash VARCHAR(255) NOT NULL COMMENT '密码哈希值',
    full_name VARCHAR(100) COMMENT '全名',
    avatar_url VARCHAR(500) COMMENT '头像URL',
    role ENUM('user', 'admin') DEFAULT 'user' COMMENT '用户角色',
    timezone VARCHAR(50) DEFAULT 'UTC' COMMENT '时区',
    language VARCHAR(10) DEFAULT 'zh-CN' COMMENT '语言',
    is_active BOOLEAN DEFAULT TRUE COMMENT '是否激活',
    is_deleted BOOLEAN DEFAULT FALSE COMMENT '逻辑删除标记',
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP COMMENT '创建时间',
    updated_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP COMMENT '更新时间',
    
    INDEX idx_username (username),
    INDEX idx_email (email),
    INDEX idx_created_at (created_at)
) ENGINE=InnoDB CHARSET=utf8mb4 COLLATE=utf8mb4_unicode_ci;
```

**字段说明**：
- `username`：用户名，用于登录和展示
- `email`：邮箱，用于注册验证和密码重置
- `password_hash`：使用bcrypt加密的密码哈希值
- `role`：用户角色，user为普通用户，admin为管理员
- `timezone`：用户所在时区，用于时间显示转换
- `language`：用户界面语言偏好

### 2. stations（台站信息表）
用户的台站设置信息

```sql
CREATE TABLE stations (
    id INT AUTO_INCREMENT PRIMARY KEY COMMENT '台站ID',
    user_id INT NOT NULL COMMENT '所属用户ID',
    callsign VARCHAR(20) NOT NULL COMMENT '呼号（业余无线电呼号规范）',
    grid_square VARCHAR(6) NOT NULL COMMENT '梅登网格（4位或6位）',
    radio_model VARCHAR(100) COMMENT '电台型号',
    antenna_model VARCHAR(100) COMMENT '天线型号',
    antenna_height DECIMAL(5, 2) COMMENT '天线高度(米)',
    qth VARCHAR(200) COMMENT '所在地点描述',
    is_primary BOOLEAN DEFAULT TRUE COMMENT '是否为主台站',
    is_deleted BOOLEAN DEFAULT FALSE COMMENT '逻辑删除标记',
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP COMMENT '创建时间',
    updated_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP COMMENT '更新时间',
    
    FOREIGN KEY (user_id) REFERENCES users(id) ON DELETE CASCADE,
    UNIQUE KEY uk_user_callsign (user_id, callsign),
    INDEX idx_user_id (user_id),
    INDEX idx_callsign (callsign)
) ENGINE=InnoDB CHARSET=utf8mb4 COLLATE=utf8mb4_unicode_ci;
```

**字段说明**：
- `callsign`：业余无线电呼号，需符合规范
- `grid_square`：梅登网格坐标（如：OL63、PL09）
- `is_primary`：标记主台站，用于默认值设置

### 3. qso_logs（通联日志表）
核心的QSO（通联）记录表

```sql
CREATE TABLE qso_logs (
    id INT AUTO_INCREMENT PRIMARY KEY COMMENT '日志ID',
    user_id INT NOT NULL COMMENT '所属用户ID',
    station_id INT NOT NULL COMMENT '使用的台站ID',
    
    -- 基本通联信息
    call_sign VARCHAR(20) NOT NULL COMMENT '对方呼号',
    qso_date DATE NOT NULL COMMENT '通联日期',
    time_on TIME COMMENT '通联开始时间(UTC)',
    time_off TIME COMMENT '通联结束时间(UTC)',
    
    -- 频率和模式
    freq DECIMAL(10, 6) COMMENT '频率(MHz)',
    freq_rx DECIMAL(10, 6) COMMENT '接收频率(MHz)',
    band VARCHAR(10) COMMENT '波段(如：20m, 15m, 40m)',
    band_rx VARCHAR(10) COMMENT '接收波段',
    mode VARCHAR(20) COMMENT '模式(如：SSB, FT8, CW)',
    
    -- 信号报告
    rst_sent VARCHAR(5) COMMENT '发送信号报告',
    rst_rcvd VARCHAR(5) COMMENT '接收信号报告',
    
    -- 对方信息
    grid_square VARCHAR(6) COMMENT '对方梅登网格',
    operator VARCHAR(50) COMMENT '对方操作员',
    qth VARCHAR(200) COMMENT '对方所在地',
    
    -- 通联质量
    qsl_sent ENUM('Y', 'N', 'R', 'I') DEFAULT 'N' COMMENT '已发送QSL卡',
    qsl_rcvd ENUM('Y', 'N', 'R', 'I') DEFAULT 'N' COMMENT '已收到QSL卡',
    eqsl_sent ENUM('Y', 'N', 'R', 'I') DEFAULT 'N' COMMENT '已发送eQSL',
    eqsl_rcvd ENUM('Y', 'N', 'R', 'I') DEFAULT 'N' COMMENT '已收到eQSL',
    lotw_sent ENUM('Y', 'N') DEFAULT 'N' COMMENT '已提交到LOTW',
    lotw_rcvd ENUM('Y', 'N') DEFAULT 'N' COMMENT '已在LOTW确认',
    
    -- 扩展信息
    distance INT COMMENT '距离(公里)',
    comment TEXT COMMENT '备注',
    is_deleted BOOLEAN DEFAULT FALSE COMMENT '逻辑删除标记',
    
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP COMMENT '创建时间',
    updated_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP COMMENT '更新时间',
    
    FOREIGN KEY (user_id) REFERENCES users(id) ON DELETE CASCADE,
    FOREIGN KEY (station_id) REFERENCES stations(id) ON DELETE RESTRICT,
    INDEX idx_user_id (user_id),
    INDEX idx_qso_date (qso_date),
    INDEX idx_call_sign (call_sign),
    INDEX idx_band (band),
    INDEX idx_mode (mode),
    INDEX idx_user_date (user_id, qso_date)
) ENGINE=InnoDB CHARSET=utf8mb4 COLLATE=utf8mb4_unicode_ci;
```

**字段说明**：
- `call_sign`：对方呼号
- `qso_date`：通联日期（YYYY-MM-DD格式）
- `band`：波段分类（自动从freq计算或手动输入）
- `qsl_*`：QSL卡状态（Y=已发送/已确认，N=未发送，R=已请求，I=忽略）
- `distance`：计算所得的地理距离

### 5. statistics（统计数据表）
预计算的统计数据，用于快速查询

```sql
CREATE TABLE statistics (
    id INT AUTO_INCREMENT PRIMARY KEY COMMENT '统计ID',
    user_id INT NOT NULL COMMENT '用户ID',
    
    -- 基础统计
    total_qso INT DEFAULT 0 COMMENT '总通联数',
    total_dxcc INT DEFAULT 0 COMMENT 'DXCC数量',
    total_waz INT DEFAULT 0 COMMENT 'WAZ数量',
    total_distance INT DEFAULT 0 COMMENT '总距离(km)',
    
    -- QSL统计
    qsl_sent INT DEFAULT 0 COMMENT '已发送QSL数',
    qsl_rcvd INT DEFAULT 0 COMMENT '已收到QSL数',
    eqsl_sent INT DEFAULT 0 COMMENT '已发送eQSL数',
    eqsl_rcvd INT DEFAULT 0 COMMENT '已收到eQSL数',
    lotw_confirmed INT DEFAULT 0 COMMENT 'LOTW已确认数',

    INDEX idx_calculated_at (calculated_at)

```

## 索引优化

### 关键性能指标
- 查询1000条日志：<100ms
- 统计计算：<500ms
- 日志导入：<2s/1000条

### 索引策略
1. **主键索引**：所有表的id字段
2. **外键索引**：所有外键字段自动创建
3. **搜索索引**：
   - `qso_logs`: (user_id, qso_date), (call_sign), (band)
   - `users`: (username), (email)
   - `stations`: (user_id, callsign)
4. **范围查询索引**：date字段便于时间范围查询

## 数据一致性设计

### 事务处理
- 日志导入使用事务确保原子性
- 用户删除使用CASCADE保证数据完整性
- 统计数据更新使用排他锁

### 触发器应用
```sql
-- 日志修改时自动更新统计数据
CREATE TRIGGER update_statistics_after_log_change
AFTER INSERT, UPDATE, DELETE ON qso_logs
FOR EACH ROW
BEGIN
    UPDATE statistics SET calculated_at = NOW() 
    WHERE user_id = IFNULL(NEW.user_id, OLD.user_id);
END;
```

## 性能优化建议

### 查询优化
1. 使用EXPLAIN分析查询执行计划
2. 避免N+1查询，使用JOIN或IN操作
3. 分页查询大数据集
4. 缓存热点查询结果

### 存储优化
1. 定期归档历史数据
2. 压缩旧日志文件
3. 清理临时文件和缓存
4. 数据库备份策略（日备份、周备份）

## 备份与恢复

### 备份策略
- **日备份**：每天凌晨执行完整备份
- **周备份**：每周备份保留至少4周
- **月备份**：每月备份保留至少12个月
- **增量备份**：每6小时执行一次增量备份

### 恢复流程
1. 验证备份文件完整性
2. 恢复到临时数据库验证
3. 执行数据一致性检查
4. 切换到恢复数据库

## 扩展性考虑

### 表分片策略
当数据量超过100M行时考虑：
- 按user_id进行表分片
- 按qso_date进行时间分片
- 使用分布式数据库如MariaDB分片

### 缓存层
- Redis缓存热点查询
- 缓存失效策略：TTL + 事件驱动
- 缓存穿透保护

## SQL变更管理

使用Alembic管理数据库版本：
```bash
# 创建迁移脚本
alembic revision --autogenerate -m "描述"

# 执行迁移
alembic upgrade head

# 回滚迁移
alembic downgrade -1
```

## 数据安全

1. **加密存储**：敏感字段使用字段级加密
2. **访问控制**：数据库用户权限最小化
3. **审计日志**：记录重要操作（更新、删除）
4. **备份加密**：备份文件使用AES-256加密
