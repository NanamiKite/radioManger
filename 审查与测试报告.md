### 1.1 Electron 安全审查（优先级最高）

审查文件：`frontend/electron/main.js`、`frontend/electron/preload.js`、`frontend/package.json`、`frontend/capacitor.config.ts`

#### 🟢 良好（main.js 安全配置正确）
- `frontend/electron/main.js:25-29` —— `webPreferences` 配置**完全正确**：
  - `contextIsolation: true` ✅
  - `nodeIntegration: false` ✅
  - 使用独立 `preload.js` ✅
- `main.js:33-40` —— 内容加载策略正确：生产加载本地 `dist/index.html`（`loadFile`），仅开发期加载 `localhost:5173`。未加载任意远程内容。✅

#### 🟡 preload.js 暴露的 API 为空，但引入了 ipcMain（异常）
- `frontend/electron/preload.js:1` —— `const { contextBridge, ipcMain } = require('electron')`：在 **preload（渲染侧）** 引入了 `ipcMain`。`ipcMain` 属于主进程模块，在 preload 里毫无用处。
  - 修复：移除 `ipcMain` 导入。preload 如需 IPC，应使用 `ipcRenderer`，并通过 `contextBridge.exposeInMainWorld` 白名单式暴露，例如：
    ```js
    const { contextBridge, ipcRenderer } = require('electron')
    contextBridge.exposeInMainWorld('electronAPI', {
      saveLog: (data) => ipcRenderer.invoke('save-log', data)
    })
    ```
- `preload.js:3-5` —— 当前 `exposeInMainWorld('electron', {...})` 暴露的是**空对象**，说明 Electron 与渲染层的桥接尚未真正实现。功能缺失（非安全问题），后续实现时务必走白名单方式，**不要把 `ipcRenderer` 整个暴露**。

#### 🔴 Capacitor 配置允许明文流量（严重）
- `frontend/capacitor.config.ts:8-9` —— `androidScheme: 'http'` + `cleartext: true`：
  - `androidScheme` 应为 `'https'`，`cleartext` 应为 `false`（或省略）。
  - 当前配置允许 Android WebView 走 HTTP 明文，中继网络下 API 请求可被嗅探/篡改。
  - 修复：`androidScheme: 'https'`，删除 `cleartext: true`。若后端确为 HTTP（局域网内），至少应在 `allowNavigation` 里限定具体域名而非 `'*'`。
- `capacitor.config.ts:11` —— `allowNavigation: ['*']` 允许导航到任意域名，过于宽松，建议限定为已知后端域名。

#### 🟡 electron-builder 配置
- `package.json:30-70` —— 多平台打包配置（win nsis/portable、linux AppImage/deb、mac dmg）结构正确。
- `package.json:37-39` —— `files` 包含 `electron/**/*`，未显式设 `asar`（默认 `true`）。✅ 但建议显式声明 `asar: true` 以防默认变更。

#### 🟢 良好（鉴权覆盖与数据隔离扎实）
- 所有用户数据端点均挂 `Depends(get_current_user)`：`logs.py`、`stations.py`、`locations.py`、`shortcuts.py` 全部端点都已鉴权 ✅

- `log_service.py:11-16` —— 排序字段白名单 `SORTABLE_FIELDS`，防 SQL 注入式的动态排序 ✅（`logs.py:74-75` 配合 `pattern` 校验）
- 分页有上限：`logs.py:73` `page_size: int = Query(20, ge=1, le=100)` ✅
- `log_service.py:108` —— 呼号搜索用 `ilike`，大小写不敏感，呼号搜索体验好 ✅

#### 🔴 速率限制中间件定义了但未注册（严重）
- `middleware/rate_limit.py` 定义了 `RateLimitMiddleware`，但 `main.py:32-34` 只注册了 `CORSMiddleware`、`ErrorHandlerMiddleware`、`LoggingMiddleware`，**没有注册限流中间件**。
- 后果：**当前生产环境没有任何速率限制**，登录/注册端点可被暴力破解，任意端点可被 DoS。
- 修复：在 `main.py` 添加 `app.add_middleware(RateLimitMiddleware, max_requests=100, window_seconds=60)`（注意中间件执行顺序）。

#### 🔴 测试账号在生产默认配置下开启（严重）
- `config.py:74` —— `ENABLE_TEST_ACCOUNT: bool = True`（默认值是 True）。
- `config.py:75-77` —— 默认创建 `admin / admin123` 测试账号。
- `docker-compose.yml:34-38` —— backend 容器的环境变量里**没有设置 `ENABLE_TEST_ACCOUNT=false`**，所以 Docker 部署会用默认值 True，创建 admin/admin123。
- 后果：局域网/云部署后，任何人都能用 `admin/admin123` 登录，**等同后门**。
- 修复：`docker-compose.yml` 的 backend environment 加 `ENABLE_TEST_ACCOUNT: "false"`；同时建议把 `config.py` 默认值改为 `False`（更安全的默认）。

#### 🔴 docker-compose 中密钥为明文占位（严重）
- `docker-compose.yml:8-11` —— MySQL `MYSQL_ROOT_PASSWORD: root123`、`MYSQL_PASSWORD: password123` 明文。
- `docker-compose.yml:35` —— `DATABASE_URL` 含明文密码 `password123`。
- `docker-compose.yml:37` —— `SECRET_KEY: your-super-secret-key-change-this-in-production` 是占位符，**若直接用于生产，JWT 可被伪造**。
- 后果：按当前 compose 直接部署到公网 = 数据库密码已知、JWT 密钥已知。
- 修复：生产用 Docker secrets 或 `.env` 注入真实强随机值，compose 里用 `${VAR}` 引用。**本报告不展示真实密钥，仅指出占位符风险。**

#### 🟡 shortcuts 用进程内存存储，无持久化
- `api/v1/shortcuts.py:23` —— `_user_shortcuts: dict = {}` 是模块级全局变量，存内存。
- 后果：服务重启后用户自定义快捷链接全部丢失；多 worker 部署（`config.py:12` `WORKERS: int = 4`）时，不同 worker 的数据不一致。
- 修复：持久化到数据库（已有 shortcuts 表设计意图，见 `doc/02_数据库设计说明.md`）。

#### 🟡 异常处理暴露内部信息
- `dependencies.py:95-99` —— 认证失败时 `detail=f"Authentication failed: {str(e)}"`，把异常字符串返回给客户端。
- `logs.py:67-68` —— `except Exception as e: ... detail=str(e)` 同样暴露内部错误。
- 后果：生产环境可能通过错误信息泄露内部结构（SQL 错误、文件路径等），辅助攻击。
- 修复：生产模式（`DEBUG=false`）返回通用错误信息，详细错误只进日志。

#### 🟡 CORS 配置宽松
- `main.py:32` —— `allow_methods=["*"]`、`allow_headers=["*"]`，且 `allow_credentials=True`。
- 配合 `config.py:40-44` 的 `CORS_ORIGINS` 含多个 localhost，开发期可接受，但生产应收紧为具体域名 + 限制 methods/headers。

### 1.3 双库（SQLite/MySQL）兼容审查

审查文件：`backend/app/config.py`、`database/base.py`、`services/log_service.py`、`models/*.py`

#### 🟢 良好（双库切换机制正确）
- `config.py:26-31` —— `ACTIVE_DATABASE_URL` 按 `DATABASE_MODE` 路由，逻辑清晰 ✅
- `database/base.py:6-22` —— 引擎参数按模式区分（SQLite `check_same_thread=False`；MySQL 连接池 `pool_pre_ping`/`pool_recycle=3600`），与设计文档一致 ✅
- `database/base.py:24-31` —— SQLite 自动 `PRAGMA journal_mode=WAL` + `PRAGMA foreign_keys=ON` ✅
- 未发现手写方言专有 SQL，查询都走 SQLAlchemy ORM ✅

#### 🔴 Alembic 迁移未启用 + create_all 不改表（严重）
- `main.py:22` —— `Base.metadata.create_all(bind=engine)`：只能创建**不存在**的表，**无法修改已有表结构**（加列、改类型、加索引）。
- 项目 `requirements.txt` 有 `alembic` 但无 `alembic.ini`、无 `versions/`，未实际配置。
- 后果：
  - 本地 SQLite：改模型后需手动删 `.db` 重建（丢数据）。
  - **生产 MySQL：改模型后线上表结构不会变**，会引发运行时错误。这是最大的运维隐患。
- 修复：正式启用 Alembic（`alembic init`，`env.py` 读 `ACTIVE_DATABASE_URL`，生成 baseline + 后续迁移脚本）。

#### 🟡 VARCHAR 长度约束在 SQLite 下不生效
- 模型用 `String(50)` 等，SQLite 会**忽略长度**（字符串无上限），MySQL 才强制。
- 后果：依赖 DB 层挡超长输入的逻辑在 SQLite 本地不生效。若 Pydantic schema 没加 `max_length`，本地能存超长字符串，迁到 MySQL 报错。
- 修复：所有有长度限制的字段在 Pydantic schema 层加 `Field(max_length=N)`，不依赖 DB。

#### 🟡 ilike 在两库行为差异
- `log_service.py:108` —— `.ilike(f"%{call_sign}%")`：SQLite 的 `ilike` 实际是模拟的，MySQL 的 `ilike` 在不同 collation 下行为可能不同。
- 这是已知差异点，建议测试覆盖（见测试报告第三部分），确保呼号搜索在两库下行为一致。
